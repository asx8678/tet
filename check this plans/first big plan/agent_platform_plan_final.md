# Ultimate Build Plan for the Agent Platform (Final)

## 1. Executive recommendation
Build the product as a **dual-surface operator console**:
- **CLI is first-class** for speed, SSH/headless workflows, diagnostics, repair, and power users.
- **Phoenix LiveView is first-class** for approvals, graph inspection, active models, artifacts, timelines, and remote coordination.

Do **not** make either surface a thin wrapper around runtime internals.
Both must go through:
1. **application services** for use-cases
2. a **runtime contract** for live execution
3. a **durable domain layer** for source-of-truth state

The backbone is:
- **single Phoenix/Ecto/Postgres app first**
- **custom domain layer always**
- **native OTP runtime first**
- **formal policy/hook registry with deterministic phases**
- **optional Jido backend later, only if native runtime pain is proven**

That gives you the strongest architecture without prematurely committing to framework gravity or UI fragmentation.
It also gives you a place to adopt the best Swarm ideas—hook ordering, task/category guardrails, blocked-attempt capture, and findings pipelines—without importing weaker choices such as file-backed truth or global singleton task state.

## 2. Resolved design decisions

### 2.1 Control surface stance
Use a **dual-surface product**.

- The **CLI** is the best daily control surface for terminal-native work, remote SSH work, repair flows, headless automation, and fast operator control.
- The **web UI** must arrive early, because graph inspection, approvals, active model visibility, cost/budget display, artifact browsing, and remote coordination are much stronger there.

Final rule:
- **CLI owns speed, repair, SSH/headless, and terminal automation**.
- **LiveView owns approvals, graph inspection, active model visibility, artifact browsing, and visual inspection**.
- **Both are equal citizens because they share application services and cover the same core workflows; parity is workflow-level, not widget-for-widget mirroring**.


### 2.2 App shape
Start as a **single Phoenix app with contexts**, not an umbrella.

Why:
- it reduces early complexity
- it gets LiveView working immediately
- it avoids premature release topology decisions
- it still allows clean separation if you enforce boundaries through contexts, application services, and runtime contracts

Later, extract to an umbrella or split services only if:
- compile times become painful
- deployment topology truly needs it
- runtime/worker release separation becomes necessary
- more than one team needs harder physical boundaries


### 2.3 Persistence choice
Use **Postgres + Ecto from day one** on the mainline.

Do not build a JSON or SQLite mainline first.
The product depends too heavily on:
- durable sessions
- tasks
- approvals
- artifacts
- event streams
- cost/budget records
- graph projections
- later Oban jobs

A throwaway local prototype can still use lighter storage, but the real roadmap should not.


### 2.4 Runtime boundary
Use a **command/event/snapshot runtime contract** behind a formal runtime behaviour.

That means the runtime is not a bag of callable internals.
It is a public boundary that exposes:
- commands for changing live execution
- events for streaming updates
- snapshots for attach/replay/inspection
- health and lifecycle controls


### 2.5 Modes and discipline
Use **three hard modes**:
- `plan`
- `explore`
- `execute`

Rules:
- `plan` = no tools, no side effects
- `explore` = read-only tools only
- `execute` = write tools allowed only with task + policy + approval rules

This is better than only `plan`/`execute`, because it gives repository inspection a real home.


### 2.6 Categories
Keep categories **advisory by default**, but allow a narrow set of deterministic guardrails where the value is obvious.

Use them for:
- UI labels
- analytics
- routing hints
- graph coloring
- reporting

Add only a small initial category policy layer:
- `planning` can never widen into write or mutate behavior
- `review` and `verification` default to read/diff/test behavior
- `repair` starts diagnostic-first and must explicitly escalate before mutating
- `implementation` is the normal category that can use the full `execute` path

Rule:
- categories may **narrow** allowed behavior
- categories may never **widen** what mode, task policy, approvals, or budgets already allow

This keeps most categories lightweight while still capturing the most useful discipline from deterministic task gating.

### 2.7 Event model
Use **typed event streams early** and capture **blocked or modified attempts** as first-class events.

Do not overload one catch-all "Bronze" log for operational behavior.
Instead keep separate typed records for:
- domain events
- runtime events
- policy events
- tool runs
- LLM calls
- error records

Then later build:
- **Bronze** = raw capture layer over those typed records, including blocked attempts and policy traces
- **Silver** = derived scoring, findings, and operational analytics
- **Gold** = promoted heuristics, routing rules, and profile candidates

Bronze is a replayable capture layer over durable records, not a substitute for them.

### 2.8 Policy middleware / hook system
Use a **formal policy middleware registry** in the runtime.

Recommended rules:
- hooks subscribe to specific event phases and event types
- pre-action hooks run in descending priority
- post-action hooks run in ascending priority
- hook outcomes are limited to `continue`, `block`, `modify`, or `annotate`
- deterministic rules run before any LLM-backed steering
- every block or modification produces a durable policy event and an operator-visible explanation

This adopts the strongest part of the Swarm architecture: deterministic, inspectable interception around tool execution, without letting the hook system become the owner of business truth.

### 2.9 Jido position
Jido is **optional later backend work**, not the center of the product.

Allowed Jido scope later:
- named/durable runtime instances
- hibernation/thaw
- worker lifecycle management
- runtime observability
- runtime-only metadata plugins

Forbidden Jido scope:
- prompt composition ownership
- tasks/modes/approvals policy ownership
- artifacts/plans/errors as source of truth
- config/secrets/provider routing
- maintenance-hour learning pipeline

## 3. Product stance
You are building an **Elixir-native agent platform** for coding and operator-guided autonomous work.

It should eventually support:
- saved provider/model configuration
- secure API key and remote credential handling
- strong prompt refinement before dispatch
- safe local and remote workspace tools
- tasks, approvals, budgets, mode enforcement, and deterministic policy traces
- a universal worker runtime with dynamic profiles
- planner/executor separation
- orchestrator-driven workflows
- durable plans, artifacts, transcripts, and event streams
- active model visibility, graph inspection, and policy-trace inspection
- remote Linux/VPS execution
- structured diagnostics and repair flows
- a nano repair path when the main app will not boot
- maintenance windows that improve heuristics, profile candidates, and reusable findings from evidence

The product should feel like a **disciplined operator console**, not a toy chat app and not a fake IDE.

## 4. Non-negotiables

### 4.1 Runtime and policy
- The runtime must **enforce** policy; prompts only **describe** policy.
- Domain truth must never live only in a running process.
- Autonomous writes always require a task.
- Every tool attempt, including blocked attempts, must emit durable events.
- Deterministic hard gates must run before any LLM steering or reminder logic.
- `plan` mode is read-only and tool-free.
- `explore` mode allows only read-only tools.
- `execute` mode requires task + policy + approval where applicable.
- Processes are disposable; durable state lives in storage.
- Errors become repair work, not just terminal noise.

### 4.2 Architecture
- CLI and LiveView never talk directly to concrete runtime internals.
- Application services are the public surface for use-cases.
- UI parity lives at the application-service and workflow level, not at the widget level: CLI and LiveView share application services and cover the same core workflows via native UX idioms, while each surface keeps its strengths (CLI for speed, repair, SSH/headless, and terminal automation; LiveView for approvals, graph inspection, active model visibility, and artifact browsing).
- Use one universal worker runtime instead of many hardcoded agent types.
- Use one model settings funnel and keep provider quirks centralized.
- Use a named-stage prompt pipeline, not a monolithic system string.
- Use your own agent loop, tool middleware, and policy middleware; do not depend on framework monkey patches.
- The hook system may influence execution but must not become the durable source of truth.
- Treat remote workers as execution infrastructure, not as the product’s main control plane.
- Keep self-improvement offline and evidence-driven.


### 4.3 Stage-gate discipline
- Use explicit go/no-go gates as anti-complexity checkpoints; crossing a gate requires a reasoned decision backed by evidence, not a vibe.
- Do not cross Stage 4 -> 5 and enable write tools until read-only tool coverage is proven, the `ExecutionTarget` abstraction is stable, and the approval skeleton has been exercised end-to-end.
- Treat the existing Stage 6 native-agent-loop gate as part of this discipline; do not build orchestration until the single-agent UX is genuinely good.
- Do not cross Stage 8 -> 9 and add Oban until durable-execution need is measured, simpler approaches have actually failed, and operator pain is documented.
- Require every gate review to produce a decision artifact that cites evidence; no silent stage advancement.


### 4.4 UX
- Always show current provider/model.
- Always show current mode.
- Always show active task when one exists.
- Always show execution target.
- Always show why an action was blocked.
- Make policy decisions inspectable, not magical.
- Separate streamed reasoning, tool activity, and final answer text.
- Do not hide model swaps, fallbacks, or approval requirements.

---


### 4.5 Deterministic invariants
- No mutating tool without an active task in an allowed mode (enforced per §10.1, §10.2, §10.3).
- Mode and category can only narrow permissions; they can never widen them (governed by the category narrow-only rule in §10.3 and the hook contract in §6.1.1 / §12.6.1).
- Blocked attempts are captured just like successful execution attempts (see `ToolBlocked` in §2.7 and the `phase` / `block_reason` / `policy_trace_artifact_id` fields in §11.5 and the `tool_runs` entity in §7.1).
- Task state changes should emit short next-step guidance so the runtime does not thrash (see the task guidance pattern in §10.2 and first-action primers in §12.6.5).
- Gold findings or profile candidates never auto-promote into live behavior without review (enforced by the review queue in §14.4 findings pipeline).

#### Event versioning
Every durable event carries both `event_type` and `event_version`.

- Versioning policy is **additive-first**: new fields are appended; existing fields are not removed or re-typed within a version.
- When shape changes are not backward-compatible, introduce a new `event_version` and provide **upcasters/adapters at read boundaries** (projections, replay, consumers) rather than mutating stored history.
- Runtime contract tests should **pin event compatibility** where practical — a consumer built for version *n* must tolerate version *n+1* additive fields without breaking.
- The dual-write discipline in §6.3 already guarantees that canonical state and event envelopes are emitted atomically; versioning does not relax that constraint.
- Upcasters live at the consumer boundary, never inside the write path.

#### 6.1.1 Policy hook behaviour
Define a small formal hook behaviour early.

Each hook should declare:
- `name`
- `phase` (`message`, `provider`, `tool_before`, `tool_after`, `lifecycle`)
- `priority`
- `filter`
- `handle`

Allowed outcomes:
- `continue`
- `block`
- `modify`
- `annotate`

Rules:
- pre-action phases execute high → low priority
- post-action phases execute low → high priority
- deterministic hooks run before LLM steering hooks
- observation hooks never override hard blocks
- hook decisions are emitted as policy events rather than hidden in ephemeral logs
- hooks return decisions to the runtime; they do not become the owner of durable business state

### 6.2 Application service contract
Use application services for all use-cases.
Examples:
- create session
- submit prompt
- accept refined prompt
- create/continue task
- request/resolve approval
- switch execution target
- attach to session
- fetch graph snapshot
- fetch active models
- fetch policy trace
- fetch findings queue
- run diagnostics

The services orchestrate domain + runtime work. UIs should never stitch those flows together themselves.

### 6.3 Dual-write discipline
- Writing canonical state and event envelopes separately creates drift between the durable record and the emitted history. That is a non-negotiable durability risk.
- Every state-changing write MUST insert or update the canonical row and emit the corresponding event envelope inside a single `Ecto.Multi` transaction.
- No exceptions. No deferred emit after commit. No best-effort publish. No event-only write path. No canonical-only write path for state-changing operations.
- Canonical tables are truth. The event log is audit and telemetry.
- Read queries hit canonical tables for correctness. They never replay the event stream to recover truth.
- Projections are materialized views, not event-sourced aggregates.
- Graph, timeline, and active-model projections are rebuilt from canonical state plus targeted event subscriptions.
- Projections are disposable and rebuildable.
- Consequences:
  - there is no event-sourced purity at the core
  - CQRS/ES is explicitly rejected as the system-of-record model
  - projections may lag, but they never redefine or diverge from truth
  - a broken event consumer can degrade observability, never corrupt durable state
- Enforcement:
  - application services own write paths
  - reviewers must reject any path that emits an event without writing canonical state
  - reviewers must reject any path that writes canonical state without emitting the corresponding envelope

### 7.3 Artifact rules
- immutable
- referenced by durable ids
- content-addressed by hash where practical
- stored in DB for small content and external/blob/file storage for large content
- chunk-readable for large logs and diffs

Messages and turns hold human-facing summaries and references.
Artifacts hold the real content.

#### Storage backend defaults
- **Content hash**: SHA-256. Every artifact is content-addressed by its SHA-256 digest.
- **Inline threshold**: payloads ≤ ~64 KB may live inline in the database row. This avoids an extra hop for small tool outputs and prompt fragments.
- **Large payloads**: stored externally in a blob store, referenced by the artifact row.
- **Dev/local default**: local-filesystem artifact store (a well-known directory under the app's data path, keyed by date/session/hash).
- **Production default**: S3-compatible or MinIO-compatible blob store.
- **Path layout**: deterministic and inspectable — e.g. `<date>/<session_id>/<hash_prefix>/<hash>`. Operators should be able to browse the store without a query layer.
- **Chunk readability** is preserved for large logs and diffs regardless of backend; blob objects are laid out so that range reads or line-oriented streaming work without downloading the entire payload.

## 8. Model/provider architecture


### 8.1 Model registry
Recommended precedence:
1. bundled defaults
2. local extension catalog
3. provider/plugin overlays
4. user overrides
5. session overrides
6. request overrides

Later merges win.


### 8.2 Provider identity vs transport
- **provider identity**: Anthropic/OpenAI/Gemini/etc behavior family
- **transport adapter**: how the request is actually sent

This is critical for:
- reasoning/interleaving support
- tool call format
- unsupported parameter removal
- prompt adaptation
- retry/fallback policy


### 8.3 Typed settings scopes
- global defaults
- provider-family defaults
- model defaults
- workspace overrides
- session overrides
- task overrides
- per-run overrides

Do not flatten them into lossy string maps.


### 8.4 Model settings funnel
It should handle:
- provider quirks
- unsupported parameter stripping
- thinking/reasoning settings
- token limits
- timeout/retry policy
- fallback compatibility
- pricing metadata exposure


### 8.5 Capability schema
- supports streaming
- supports tools
- supports structured output
- supports reasoning mode
- supports interleaved reasoning
- max context
- recommended defaults
- pricing metadata
- fallback compatibility

Do not rely mainly on model-name heuristics.


### 8.6 Fallback & rate limiting
- **Capability-aware fallback chains**: each model entry may declare an ordered fallback list. Before falling back, verify the candidate supports the required capabilities (tools, structured output, reasoning, context window, etc.). A fallback that silently drops tool support is worse than a clear failure.
- **Cost-aware fallback**: prefer cheaper or lower-latency alternatives where multiple candidates satisfy capability requirements, but never hide the swap from the operator.
- **Rate limits**: per-provider and per-model, enforced via token-bucket or equivalent. Limits should be configurable per scope (global, provider-family, model).
- **Circuit breaker**: after repeated provider failures within a window, trip a circuit breaker that short-circuits to the next fallback or returns a clear error. Avoid hammering a degraded provider.
- **Operator visibility**: every fallback event must be logged, emitted as a runtime event, and visible in the active-model rail (§13.6) and session timeline. Never hide a model/provider swap from the operator.
- **Compatibility checks**: the model settings funnel (§8.4) must validate that a fallback candidate satisfies the current call's constraints before routing to it.


### 8.7 Modern LLM features
- **Prompt caching** (Anthropic/OpenAI-style): where available, structure prompts so cacheable prefix segments align with the provider's cache-control hints. Cache hits reduce cost and latency; cache-miss accounting should feed into cost/budget tracking (§10.5).
- **Reasoning / extended-thinking token streams**: treat reasoning tokens as a first-class stream type within the existing event model (e.g. `ReasoningChunkStreamed`). They count toward token budgets and cost but are distinct from output tokens. Compaction (§12.8) may summarize reasoning streams differently from output.
- **Parallel tool calls**: where a provider supports multiple tool calls in a single response, the runtime should dispatch them concurrently while preserving per-tool-run event discipline. Results are collected before the next model turn.
- **Structured outputs / schema-constrained responses**: when a call requests a structured output, the capability schema (§8.5) must confirm provider support. The settings funnel strips the constraint on fallback to a provider that lacks it, and the operator is notified.
- **Multimodal inputs** (images, PDFs, future modalities): treat them as artifacts in the existing artifact flow. The prompt composer attaches them by reference; the provider adapter translates references into the provider's multimodal format. This keeps multimodal out of the prompt string itself.
- **MCP (Model Context Protocol)**: treat MCP servers as a **tool source / adapter**, not a replacement for the platform's own tool registry (§11.2) or policy model (§10.4). MCP-sourced tools are registered in the platform tool registry with the same capability declarations and approval constraints as any other tool. MCP is one input channel, not a parallel authority.

### 9.5 Prompt Lab
Prompt Lab is a first-class feature, not a toy add-on.

It should help the operator inspect and improve prompts, but it must stay inside the trust boundary used elsewhere in this system, including workspace rules trust (§9.2).
Use epistemic humility: static heuristics can inform judgment, but they do not outrank human judgment, and they do not override runtime policy or operator intent.

It should:
- score prompt quality
- identify missing context
- recommend mode
- recommend category
- produce a conservative rewrite
- produce a more autonomous rewrite
- never execute the task itself

#### Advisory, not gatekeeping
Scores are advisory and auditable, never blocking.
Prompt Lab may warn, highlight risks, and recommend changes, but it must never refuse to dispatch a prompt on the basis of its own score.

"Send as-is" must be a first-class, always-available action.
Do not silently rewrite prompts without operator consent.

#### Explainability and history
A score without derivation is garbage.
Every score must expose:
- inputs considered
- rules fired
- weights
- heuristic name
- heuristic version

Score history for the same prompt lineage should be inspectable and browsable.
Operators should be able to compare prior scores and advisory output across revisions.

#### Output framing
Numerical scores must be labeled in UI and logs as heuristic advisories.
Do not present them as pass/fail grades.
Do not render them as a big green checkmark that implies authorization, correctness, or readiness.

#### Scoring methodology
Prompt Lab scoring is **heuristic-first**.
- Primary scores are derived from deterministic heuristics: prompt structure, ambiguity, missing context, constraint coverage, actionability, and mode/category alignment.
- An optional **LLM-based rubric pass** may be enabled behind budget/policy controls. When enabled, it refines heuristic scores with a structured rubric evaluation, but it never overrides the heuristic baseline — it supplements it.
- Heuristic inputs and their weights are versioned and exposed in score derivation metadata (see Explainability and history above).
- Cache advisory results by **prompt hash** and, where available, **prompt-lineage hash** (derived from the workspace + session + task context). A prompt that is structurally identical and in the same context should not re-score from scratch.
- Cached results are invalidated when heuristic versions change or when lineage context shifts.
- This aligns with the advisory, non-authoritative framing above: cached scores are hints, not verdicts.

Recommended outputs:
- overall score
- clarity score
- context score
- constraints score
- ambiguity score
- actionability score
- autonomy safety score
- issues
- missing questions
- recommended mode
- recommended category
- conservative rewrite
- autonomous rewrite
- score derivation metadata
- score history for prompt lineage

#### Recommended outputs
Prompt Lab should produce a structured result containing at minimum:
- overall score
- clarity score
- context score
- constraints score
- ambiguity score
- actionability score
- autonomy safety score
- issues
- missing questions
- recommended mode
- recommended category

These outputs are advisory per §9.5 (Advisory, not gatekeeping) and must be explainable per §9.5 (Explainability and history). The scoring methodology governing how each score is derived is defined in §9.5 (Scoring methodology).


### 9.6 Prompt debugging
Add prompt debugging early.
Expose:
- each prompt layer
- the source of each layer
- final merged prompt
- provider/model rewrite differences

This should exist in:
- web inspector
- CLI `prompt debug`

## 10. Modes, tasks, approvals, and budgets

### 10.1 Modes
Session has a default mode.
Task may narrow the mode, but not widen it.

Examples:
- session `explore` cannot be widened by a task into `execute`
- session `execute` can still host a task in `plan` or `explore`

#### `plan`
Allowed:
- reasoning
- planning output
- task proposals

Not allowed:
- tools
- file reads
- commands
- writes

#### `explore`
Allowed:
- read-only tools
- file reads
- search
- non-mutating commands
- repository inspection

Not allowed:
- writes
- patch apply
- mutations

#### `execute`
Allowed:
- write tools
- patch apply
- format/test flows
- completion/verification flows

Requires:
- task
- policy check
- approval routing where applicable
- budget checks


### 10.2 Tasks
Tasks are required for serious tracked work and for autonomous writes.
Do not require heavyweight manual task creation for every small operation.

Use:
- explicit operator-created tasks for meaningful work
- lightweight auto-created tasks for autonomous writes and long-running flows
- upgrade path from auto-task to named task

Recommended early task states:
- `pending`
- `in_progress`
- `blocked`
- `completed`
- `cancelled`

Rules:
- mutating work requires a task in `in_progress`
- one worker should have one active task unless the runtime explicitly supports concurrent task lanes
- task transitions should emit short guidance events such as “continue current task”, “pick the next pending task”, or “create a follow-up repair task”
- task truth lives in the domain layer; the runtime may cache active task refs, but never owns the authoritative task list

### 10.3 Categories
Recommended categories:
- planning
- research
- implementation
- review
- verification
- repair

Categories remain mostly advisory, but they should support a **small deterministic policy matrix** from the beginning.

Recommended early default behavior:
- `planning` -> read-only reasoning and inspection only; no mutating tools
- `research` -> read-only tools and repository inspection
- `implementation` -> normal execute path
- `review` -> read, diff, annotate, and test; writes only by explicit escalation
- `verification` -> tests, checks, and report artifacts; no source mutation by default
- `repair` -> diagnostic-first; writes only after explicit repair intent, approval routing, and budget checks

Rule:
- category can narrow the tool set further
- category can never widen the allowed behavior of the current mode
- deterministic category rules run before any LLM steering

### 10.4 Approval classes
Use at least three approval classes.

#### `auto`
Examples:
- model-only turns
- safe reads
- approved read-only commands

#### `soft`
Examples:
- bounded command runs
- format/test in safe contexts
- low-risk writes inside an already approved path

#### `hard`
Examples:
- file writes
- patch application
- secret access
- remote deployment
- destructive operations

Every approval should be:
- durable
- replayable
- correlated to session, task, tool run, and diff artifact
- visible in web and CLI


### 10.5 Budgets
Track budgets from the beginning:
- session token budget
- task token budget
- daily operator cost budget
- tool round-trip budget
- command timeout budget
- diff size budget
- max concurrent workers
- max remote sessions per target

Do not hide cost, token, or fallback behavior from the operator.

### 11.1 ExecutionTarget abstraction

Start with:

- SSH remote target

This prevents local-only assumptions from forcing a rewrite later.


### 11.2 Tool registry
Every tool should declare:
- name
- description
- input schema
- output schema
- read/write nature
- allowed modes
- default category posture (`read_only`, `diagnostic`, `mutating`, `verification`)
- approval class
- target capabilities required
- timeout defaults
- environment policy
- batchability / streamability hints

### 11.3 Initial read-only tools
Ship early:

- `read_file`

- `glob`
- `git_status`
- `git_diff`
- `git_log`
- `run_command_readonly` with allowlist
- repo structure summarizer

### 11.4 Initial write tools
Ship only after tasks + approvals + budgets exist:
- `write_file`
- `apply_patch`
- `create_artifact_from_diff`
- `run_tests`
- `format_code`

Treat `run_tests` and `format_code` as mutating or semi-mutating when they can change the workspace.

### 11.5 Tool result normalization
Every tool result or blocked attempt should normalize to a common structure:
- tool name
- args
- target
- phase (`proposed`, `blocked`, `executed`)
- status
- block reason if any
- stdout/stderr refs
- changed files
- timings
- exit code
- artifact refs
- policy trace refs
- finding tags
- correlation ids

Keep the common envelope thin:
- tool id/name
- tool call id
- args or arg summary
- target
- status
- duration_ms
- cost_tokens where applicable
- exit code where applicable
- correlation ids
- artifact refs

Treat artifact refs as a bag keyed by payload role.
Examples:
- `stdout`
- `stderr`
- `diff`
- `output`
- `report`

Do not inline non-trivial payloads into the tool result:
- stdout/stderr belong in artifact rows, referenced by id
- diffs belong in artifact rows, referenced by id
- large listings and generated reports belong in artifact rows, referenced by id
- changed files may be summarized inline by path, but full diff content stays in artifacts

Trivial read tools may return minimal JSON instead of a padded envelope.
Examples:
- `cat`
- `stat`
- small file reads

Do not force trivial reads into a universal schema full of empty fields.
Avoid the anti-pattern of one fat schema with twenty optional fields.
Prefer:
- small shared header
- minimal tool-specific body for trivial cases
- artifact references for anything large or non-trivial

### 11.6 Command environment hardening
From the start:
- scrub environment vars
- enforce workspace root policy
- timeouts
- kill runaway child processes
- capture stdout/stderr separately
- avoid shell interpolation where possible
- keep allowlists explicit
- distinguish diagnostic commands from mutating commands in registry metadata

### 11.7 Post-tool analyzers
Run a small analyzer chain after tool completion.

Recommended analyzers:
- **failure analyzer** for stderr, exit-code failures, repeated write problems, and repair-task candidates
- **verification analyzer** for missing tests, missing docs, or follow-up verification work
- **insight analyzer** for reusable recipes, file relationships, and noteworthy outputs
- **efficiency analyzer** for waste patterns such as repeated grep loops, context pressure, or obvious batching opportunities
- **budget analyzer** for unusually expensive retries, long tool loops, or timeout churn

Rules:
- analyzers emit structured findings or events
- analyzers are asynchronous where possible
- analyzers may recommend tasks or Gold candidates, but they do not directly mutate live policy

## 12. Runtime kernel and agent loop

### 12.1 Native OTP runtime first
Build the first runtime using:
- `GenServer`
- `DynamicSupervisor`
- `Registry`
- PubSub
- Presence where useful

Do not add Jido to the critical path early.


### 12.2 Session runtime
One runtime per interactive session.
State should remain small and disposable:
- session id
- current mode
- execution target ref
- active task ref
- transcript/event cursors
- active worker refs
- active model call refs
- pending approvals
- usage counters
- ephemeral caches/cursors/timers


### 12.3 Universal worker runtime
Do not hardcode many separate agent modules.
Use one universal runtime that can act as:
- planner
- retriever
- reviewer
- critic
- coder
- repairer
- prompt refiner

based on profile data.


### 12.4 Native agent loop
A turn should roughly do:
1. assemble prompt + context
2. call model
3. classify streamed/result parts
4. if terminal answer -> finish
5. if tool intent/calls -> route through the hook manager and policy chain
6. persist policy decisions, results, artifacts, and events
7. compact history if needed
8. continue until terminal answer or usage limit

### 12.5 Response classifier
Support distinct classes:
- terminal text answer
- structured output answer
- tool call batch
- approval-needed action
- steering-needed action
- retryable provider/runtime error
- fatal error

### 12.6 Middleware and hook lifecycle
Use a formal hook manager with explicit phases, priorities, and filters.

Use a precise lifecycle, not vague “hooks everywhere”.
The recommended high-level sequence is:
1. **normalizers/enrichers before validation**
2. **validation**
3. **policy hooks after validation**
4. **execution**
5. **completion hooks**

Use hooks at meaningful boundaries:
- stream started
- stream ended
- tool proposed
- tool completed
- answer finalized

Do not run arbitrary business hooks on every chunk.

#### 12.6.1 Hook contract
Each hook should declare:
- name
- phase
- priority
- filter
- decision handler

Allowed decisions:
- `continue`
- `block`
- `modify`
- `annotate`

Hooks do not own durable state directly.
They return decisions to the runtime, which emits events and calls domain services.

#### 12.6.2 Execution order
Use deterministic ordering:
- pre-action phases run **descending** priority
- post-action phases run **ascending** priority
- blocked attempts still emit tool-intent and policy events
- observational hooks never override earlier hard blocks

Recommended phases:
- `message`
- `provider`
- `tool_before`
- `tool_after`
- `lifecycle`

#### 12.6.3 Recommended priority bands

> **⚠️ Deferral marker (→ Stage 6)**
> The full priority band enforcement (bands 100–0) lands at Stage 6. Bands 100–90 (security) and 89–80 (approval/budget) MUST be enforced from Stage 4/5. Bands 79–70 (LLM steering) and 69–50 (task guidance) are reference-only until Stage 6 and require the steering evaluator (§12.6.4). Tracing/metrics at 49–0 should ship from Stage 4 as observability, not as policy gates.
>
> The general ordering rule — deterministic rules always run before model-backed steering — applies from Stage 4 onward regardless of enforcement depth.

- **100–90:** security, workspace trust, mode/task/category hard gates
- **89–80:** approval, budget, rate-limit, repeated-write validation
- **79–70:** optional LLM steering and relevance guidance
- **69–50:** task guidance, post-acting reminders, verification nudges
- **49–0:** tracing, metrics, Bronze capture, findings emitters

Deterministic rules always run before model-backed steering.

#### 12.6.4 Steering evaluator

> **⚠️ Deferral marker (→ Stage 6)**
> The LLM-backed steering evaluator lands at Stage 6 as part of the mature hook manager. Until Stage 6, hooks with priority < 70 are NOT required. The lightweight first-action primers (§12.6.5) cover the minimum guidance case for Stages 4–5. Enabling steering before Stage 6 risks LLM-based policy decisions running before the deterministic gate stack has been proven stable.

Use an optional LLM-backed steering evaluator only after hard deterministic checks pass.

Recommended steering outcomes:
- `continue`
- `focus`
- `guide`
- `block`

Recommended steering context:
- proposed tool name and arguments
- active task title, mode, and category
- recent completed or blocked tasks
- current plan or active plan summary
- recent conversation turns
- recent artifacts and findings
- active execution target
- recent tool history and budget state

LLM steering should be reserved for ambiguous, costly, or drift-prone actions.
It is an assistive layer, not the primary policy engine.

#### 12.6.5 First-action primers and task guidance
Adopt two lightweight guidance patterns:
- on the **first tool in `explore`**, optionally inject a short repo-inspection or decomposition reminder
- after **TaskCreate** or meaningful **TaskUpdate**, emit a short next-step guidance event

These reduce wasted loops without weakening deterministic policy.

### 12.7 Usage limits and resumption
Add hard caps early:
- max model↔tool round trips per turn
- max retries/backoff
- max concurrent workers

If limits are reached, stop with a resumable reason instead of silently looping.


### 12.8 Compaction and pruning
Implement compaction as a first-class subsystem.

Never compact away:
- product constitution
- current mode
- active task summary
- approval state
- latest constraints
- identity block

Strategies:
- naive truncation
- summary artifact + replacement reference
- provider-aware compaction

Also add:
- interrupted tool-call pruning
- history shape validation before next model request

#### Compaction triggers
Compaction should fire on well-defined triggers, not just ad-hoc operator action:
- **Token/context threshold**: when the accumulated token count approaches the model's context window, trigger compaction before the next model call.
- **Turn count threshold**: when turn count exceeds a configurable ceiling for the current session/task, compact to preserve context headroom.
- **After large tool batches or artifact-heavy steps**: a single turn that produces multiple tool runs or large artifacts can saturate context; compact after such steps.
- **Operator-triggered compaction**: allow explicit `compact` commands from CLI and web UI for diagnostics or manual context shaping.
- **Pre-flight validation**: before dispatching the next model call, validate that the compacted context still satisfies the "never compact away" constraints above. If validation fails, abort the call and surface the error to the operator.

The "never compact away" list (constitution, mode, active task, approval state, latest constraints, identity block) is absolute and applies regardless of trigger.


### 12.9 Crash dump policy
On abnormal runtime stop, emit a crash snapshot containing:
- session id
- current turn id
- pending tool calls
- pending approvals
- active worker refs
- last known model call
- recent event cursor

This greatly improves repair and diagnostics.


### 12.10 Stage 1 streaming reference architecture
Stage 1 streaming should optimize for **responsiveness, operational simplicity, and clear failure boundaries**.
The goal is not to build a perfect forever-streaming subsystem on day one; the goal is to ship a design that is easy to reason about, cheap to observe, and safe to evolve.

#### Session process responsibilities
Treat the session `GenServer` as a **coordinator only**.
It should own orchestration state and policy decisions, but it should **not** perform streaming transport work itself.

The session process should:
- accept `Session.submit_prompt/2`
- persist the user turn immediately
- start the model call asynchronously
- track the active call ref/worker pid
- aggregate normalized stream updates into ephemeral state
- publish coarse UI updates
- finalize the assistant turn once generation completes
- update terminal status on completion, cancel, or error

The session process should **not**:
- hold socket connections open
- parse SSE frames or provider-specific chunk formats
- perform per-token database writes
- push every raw provider event through PubSub
- block its mailbox on network or parsing work

This keeps the session runtime small, disposable, and aligned with OTP responsibilities.

#### Per-call stream worker model
Run each LLM streaming call in its own transient worker under `Task.Supervisor`.
Use **one supervised process per model call**.

Recommended shape:
- session process starts a child via `Task.Supervisor`
- the worker owns the provider request lifecycle for that one call
- the worker translates provider-native chunks into internal normalized messages
- the worker exits normally on `:done` and is discarded
- the worker exits on failure without taking down the session process

This creates a clean boundary:
- session = orchestration and state transitions
- worker = transport, parsing, normalization, and flush timing for one call

Avoid inventing a heavier abstraction until real complexity demands it.
A supervised task is enough for Stage 1 and obeys YAGNI instead of making the architecture cosplay as Kubernetes.

#### Async kickoff via `handle_continue`
`Session.submit_prompt/2` should return immediately after the user turn is accepted and the session state is updated.
Do the expensive kickoff work in `handle_continue`.

Recommended sequence:
1. client calls `Session.submit_prompt/2`
2. session validates state and rejects only obvious invalid transitions
3. user turn is persisted immediately
4. session stores a pending/streaming status in memory
5. session returns quickly to the caller
6. `handle_continue` starts the supervised stream worker
7. worker begins emitting normalized chunks back to the session

This avoids turning the session call path into a latency funnel and keeps UX snappy even when provider setup is slow.

#### Internal normalized chunk protocol
Normalize all provider-specific stream events into a tiny internal protocol before they touch session state.
For Stage 1, the stream worker should only send:
- `:text_delta`
- `:reasoning_delta`
- `:tool_delta`
- `:done`
- `:error`

Practical rules:
- `:text_delta` carries user-visible answer text
- `:reasoning_delta` carries optional reasoning/debug text if the product mode allows it
- `:tool_delta` carries structured tool-call progress fragments or assembled tool intent data
- `:done` marks normal stream completion and includes final metadata needed for persistence
- `:error` marks provider/runtime failure and includes a classified error payload

Do not leak raw SSE events, transport chunks, or provider event names across the boundary.
The rest of the system should reason about product semantics, not OpenAI/Anthropic/whatever wire trivia.

#### Batching and flush strategy
Do not forward every token-sized fragment.
Batch aggressively enough to protect the session mailbox and UI, but lightly enough to preserve responsiveness.

Stage 1 policy:
- accumulate deltas as `iodata`
- flush every **50–100 ms** or when a size threshold is crossed
- emit a final flush before `:done`
- keep batching logic in the stream worker, not the session process

Implementation notes:
- maintain separate buffers for text, reasoning, and tool payload fragments when helpful
- use monotonic time for flush scheduling
- prefer appending `iodata` over repeated binary concatenation
- send one normalized message per flushed category batch, not one message per provider fragment

This gives good first-token feel without paying the “millions of tiny messages because streaming looks cool in demos” tax.

#### Persistence policy
Persistence should reflect product value, not transport noise.

Stage 1 persistence policy:
- persist the **user turn immediately** when the prompt is submitted
- keep the in-progress assistant stream **ephemeral and in-memory** during generation
- persist the **assistant turn once** when generation completes successfully
- write throttled partial snapshots **only if evidence demands it** for recovery, compliance, or long-running reliability cases

Explicitly avoid:
- per-token writes
- per-chunk writes by default
- treating the database as a live stream buffer

Why this stance is correct for Stage 1:
- lower write amplification
- simpler transactional semantics
- easier turn reconstruction
- less DB contention under concurrent sessions
- fewer weird partially-written assistant rows after cancel/error paths

If later evidence shows snapshotting is required, add it deliberately with a clear cadence and retention policy rather than sneaking in accidental event sourcing through the side door.

#### PubSub contract
PubSub should carry **coarse-grained UI updates**, not transport chatter.

Publish:
- batched text deltas
- batched reasoning deltas where allowed
- terminal status changes such as completed/cancelled/failed

Do not publish:
- every token
- raw provider chunks
- SSE control frames
- ultra-high-frequency internal flush noise

The UI should be told what materially changed, not forced to reenact the provider socket transcript in public.

#### Cancel semantics
Cancellation should be brutally simple.
When a user cancels an active generation:
- terminate the supervised stream worker
- mark the active call as cancelled in session state
- publish the terminal cancelled status
- avoid extra stream cleanup complexity inside the session process

The session process should **not** attempt to unwind provider streaming internals itself.
Its job is only to update authoritative state after the worker dies or is terminated.

This makes ownership obvious:
- worker owns the live stream
- session owns the state machine

#### Backpressure and mailbox discipline
Assume bursts, slow subscribers, and uneven provider chunking from day one.
The session process should monitor mailbox depth and react before it becomes a tiny tragedy.

Stage 1 backpressure rules:
- watch mailbox length for the session process and, if needed, the UI broadcaster path
- when depth rises, coalesce deltas more aggressively in the worker
- increase effective batch size and/or flush interval within safe bounds
- prefer freshness over chunk fidelity
- drop unnecessary intermediate granularity before allowing mailbox growth to cascade

In practice, if the system is under pressure, it is better for the UI to see larger newer chunks than to faithfully receive every microscopic delta from 800 ms ago.

#### Day-one instrumentation
Instrument the streaming path immediately so later tuning is based on evidence instead of vibes.
Track at minimum:
- time-to-first-token
- total latency
- chunk rate
- cancel rate
- stream errors
- mailbox depth

Also useful on day one:
- active stream worker count
- flush size distribution
- completion vs cancel vs error outcomes by provider/model
- time spent from submit to worker start

These metrics should be emitted with session id/model/provider tags where appropriate and kept cheap enough to leave on by default.

#### Reference flow summary
A Stage 1 happy path should look like this:
1. client submits prompt
2. session persists the user turn and returns quickly
3. `handle_continue` starts a supervised stream worker
4. worker opens the provider stream and normalizes incoming chunks
5. worker batches `iodata` for 50–100 ms or until size threshold
6. session receives coarse deltas and updates ephemeral in-memory assistant state
7. session publishes coarse PubSub updates for UI consumers
8. worker sends `:done`
9. session persists the assistant turn once and marks the turn complete

This architecture gives fast streaming UX without coupling core session state to provider transport details, database write amplification, or per-token PubSub spam.

## 14. Eventing, projections, and analytics

### 14.1 Typed streams
Keep distinct streams for:
- domain
- runtime
- policy
- llm
- tool
- error

#### Event ordering guarantees
- Each stream provides **per-aggregate monotonic ordering** via a `seq` field. Consumers within a stream can rely on `seq` being monotonically increasing for a given aggregate, such as a session or task.
- A **global ingest timestamp or order field** may be added later for cross-stream analysis, but it is optional and must not be assumed by projection code.
- Ordering guarantees are **per-aggregate first, global only where explicitly required**. Do not impose global ordering costs on every stream.
- Projection code must **tolerate gaps, retries, and rebuilds**: missing or duplicated `seq` values should not corrupt projections. Projections are idempotent with respect to re-delivery (§6.3).
- This is event ordering for typed streams, not full event-sourcing. Projections are still derived from canonical state; the event log provides audit and telemetry, not source-of-truth rebuilds.

### 14.2 Core projections
Build projections for:
- active sessions
- active models
- graph state
- approvals queue
- cost/budget state
- timeline
- recent blocked actions / policy traces
- findings review queue

### 14.3 Bronze / Silver / Gold
Use the following interpretation:

#### Bronze
Raw capture and replayable records:
- prompts
- turns
- tool intents
- blocked and executed tool runs
- task transitions
- approvals
- policy decisions
- retries
- failures
- events

Bronze must preserve task correlation, session correlation, and block reasons.

#### Silver
Derived scoring and operational analytics:
- tool usefulness
- task success rate
- wasted calls
- retry patterns
- artifact usefulness
- plan usefulness
- approval latency
- repeated write failures
- verification gaps
- repair candidates
- context-pressure or batching opportunities

Silver is where analyzers and optional LLM scoring live.
Silver outputs are advisory until promoted.

#### Gold
Promoted long-term memory and policy candidates:
- routing rules
- heuristics
- recurring repair patterns
- profile candidates
- reusable task templates
- reusable diagnostics or verification playbooks

Do not let Gold mutate live behavior without a promotion/review path.

### 14.4 Findings pipeline

> **⚠️ Deferral marker (→ Stage 12)**
> The full staged findings pipeline (batch scoring, review queue, Gold promotion workflow) lands at Stage 12 as part of maintenance analytics. Stage 7 ships the analyzer chain (§11.7) and the findings queue (§14.2 Core projections) as Bronze-level capture. Batch scoring/clustering (step 3), the review queue UI (step 4), and Gold promotion (step 5) are deferred so the system first accumulates real findings data before we build the promotion machinery on top of it.

Use a staged findings pipeline rather than one monolithic analytics job.

Recommended flow:
1. post-tool analyzers emit findings
2. findings are correlated to session, task, tool run, and artifact ids
3. batch scoring or clustering ranks them asynchronously
4. high-value findings go to a review queue
5. approved findings become Gold candidates or repair templates

This captures the useful Bronze → Silver → Gold discipline from Swarm while keeping promotion conservative and auditable.

### 15.1 Durable execution

6. apply patch

“Explain failure” should combine:
- error records
- recent tool runs
- recent policy decisions
- recent artifacts
- recent crash state

If the main app will not boot, provide a narrow rescue path.

Capabilities should stay minimal:
- inspect latest boot failure
- read minimal config
- inspect allowlisted files
- generate small patch
- compile
- run smoke boot test


### 15.5 Remote execution
Keep the local machine as the control plane.
Use remote targets for heavy work.

Local:
- CLI
- LiveView
- approvals
- orchestration visibility
- monitoring

Remote:
- repo checkout
- heavy commands/tests
- worker execution
- long-running tasks

Remote UX requirements:
- target picker
- health panel
- visible “running remotely” state
- graph edges to execution target
- logs/events streamed back
- reconnect/sync behavior on partition


### 15.6 Control-plane vs remote-target trust boundary
Do not pretend a remote machine is trusted just because it is reachable.
Trust is asymmetric:
- the control plane is trusted
- remote targets (`ssh_remote` hosts, remote backends, worker nodes) are less trusted
- every remote execution path gets least-privilege credentials scoped per `execution_target`

Credential scoping rules:
- every credential has one owning `execution_target` row
- credentials are never shared across targets
- there is no global credential bucket for remote work
- the `execution_target` record is the ownership and audit boundary

Credentials must never leave the control plane as durable secret material.
Preferred patterns:
- issue short-lived scoped tokens from the control plane
- use SSH agent forwarding when SSH is the transport
- never copy raw SSH keys, API tokens, or provider master credentials onto remote targets

Credentials must never be rendered into prompts.
This is a hard guardrail, not operator preference.
The redactor must strip anything that looks like a credential before any payload touches an LLM boundary.

Rotation flow for target-scoped credentials:
1. issue the replacement credential
2. validate it against the target
3. flip `active` on the target to the new credential
4. revoke the old credential
5. verify with `<APP> doctor --targets`

Revocation must be immediate.
Each `execution_target` row should carry `revoked_at` and treat revocation as an operational kill switch:
- new work must not schedule against a revoked target credential
- connection pools for that target should be torn down
- in-flight remote calls should be cancelled where the transport allows it

`<APP> doctor --targets` should also run credential-hygiene checks:
- no credential older than the configured threshold
- no credential referenced by zero targets
- no plaintext credential material in logs, event payloads, or diagnostic output

This section is the control-plane/remote-target lens on the same credential substrate described in §22.
§22 defines the shared secrets-management rules.
This section defines how those same rules tighten when work crosses the remote execution boundary.

Only adopt Jido if measured runtime pain exists.
Examples:
- hibernation/thaw would materially help
- durable named runtime instances are painful to implement/maintain natively
- worker lifecycle observability is insufficient
- remote runtime ownership is clearly cleaner with Jido

- child worker lifecycle
- hibernation/thaw

- runtime observability

### 16.2 Reuse directly later
Good fits:
- `InstanceManager`
- Observe/telemetry discipline
- instance-scoped runtime pattern
- directive/effect separation inside runtime
- possibly limited runtime metadata plugins

### 16.3 Reuse conceptually only
Good ideas, but do not force them everywhere:
- command + effect separation
- checkpoint points to durable state rather than storing everything
- strategies when execution models truly diverge
- worker child tracking

### 16.4 Skip or postpone
- Pods early
- worker pools for mutable coding sessions
- generic memory/identity/thread plugins without review

### 16.5 Contract rule
If you add Jido, it must pass the same runtime contract tests as the native backend.

## 21. Auth, identity, and access model
Treat auth as a small but real platform concern from day one.
Do not bolt it on after the session/task model is entrenched.
The early product can still feel single-user, but the data model and service boundaries must already know who is acting and which account owns the state.

## 21.1 User/account concept

Extend the canonical model from §7 with two durable identity concepts:
- `accounts` = the ownership, billing, and administrative boundary
- `users` = the human actors operating inside an account

Recommended durable fields:
- `accounts`: `id`, `name`, `slug`, `status`, `settings`, timestamps
- `users`: `id`, `account_id`, `email`, `display_name`, `status`, `last_seen_at`, timestamps

Relationship to the §7 entities:
- `workspaces.account_id`
- `sessions.account_id`
- `sessions.created_by_user_id`
- `tasks.created_by_user_id`
- `approvals.resolved_by_user_id`
- actor metadata on `event_envelopes` for audit and replay context

Rules:
- a workspace belongs to exactly one account
- a session belongs to exactly one workspace and exactly one account
- the session is the operator-facing conversation container
- the account is the ownership and security boundary
- the user is the actor and audit subject
- do not make tenancy depend on deriving account scope through a chain of joins on every query; stamp `account_id` where it improves clarity, authorization, and filtering
- service APIs should accept actor-aware context such as `current_account` and `current_user`, not only naked resource ids

Later additions can extend this without changing the core shape:
- `account_memberships`
- `service_accounts`
- `personal_access_tokens`

## 21.2 LiveView session auth

Use Phoenix-standard browser session auth.

Default shape:
- signed and encrypted session cookies
- `HttpOnly` cookies always
- `Secure` cookies outside local dev
- `SameSite=Lax` or stricter by default
- CSRF protection on login, logout, and all browser form mutations
- LiveView socket auth derived from the existing browser session, not a separate ad-hoc auth path

Recommended login flow:
1. the user authenticates through the configured strategy
2. the server issues a revocable session token
3. the browser stores the session reference in the session cookie
4. HTTP requests and LiveView mounts load `current_user` and `current_account`
5. logout or revocation rotates the token and disconnects active LiveView sessions

Auth strategies should be pluggable behind one identity boundary, but the defaults should stay boring:
- default: email + password
- optional: magic link for lower-friction self-hosted access
- optional and later preferred for team deployments: OIDC

Why this order:
- password auth works for local-first and self-hosted installs without external dependencies
- magic link is a convenience feature, not a foundation
- OIDC should plug in cleanly when the product moves into team/org environments

Authorization still belongs in application services.
LiveView should load identity, assign it, and call policy-aware services.
Do not hide important authorization rules in templates or socket event handlers.

## 21.3 CLI ↔ backend auth

Use personal access tokens for CLI-to-backend authentication.
Do not invent a separate CLI-specific auth path.

PAT rules:
- tokens are issued per user, per account, and per device or label
- the raw token is shown once at creation time
- only a hash is stored server-side
- the CLI sends `Authorization: Bearer <token>`
- tokens support `last_used_at`, optional expiry, and explicit revocation
- revoked or expired tokens fail hard and clearly

Issuance paths:
- create from the web security/settings UI
- create from a local bootstrap or login flow when the CLI can talk to a trusted backend

Storage rules:
- default: OS keychain or credential manager
- fallback: config file only when keychain storage is unavailable
- config-file fallback must use strict permissions and loud warnings
- never print tokens in normal logs, traces, screenshots, or crash dumps

Local-first single-user default:
- first boot can seed one owner account and one owner user
- the local CLI can then obtain a PAT through a short bootstrap flow
- after that, the CLI talks to the local backend exactly the same way it would talk to a remote backend

That last point matters.
It keeps the transport contract DRY instead of creating one opaque local auth path and one separate remote auth path.

Remote backend:
- use the same PAT mechanism
- later browser-assisted login or device-code flows may sit on top
- the backend contract still remains token-based

## 21.4 Single-user-now / multi-user-later migration path

Design for one real account even if the early product feels like “just me on my laptop.”

From day one:
- seed a default `account`
- seed a default owner `user`
- attach `account_id` to workspaces, sessions, and other durable records that need tenancy or audit clarity
- thread actor/account context through application services
- include actor metadata in important events and approvals
- keep UI copy simple early; you do not need account switching screens before there is more than one account

This gives a clean migration later:
- add `account_memberships`
- add invite flows
- add more than one user per account
- add workspace sharing or workspace-level roles if the product proves the need
- add stronger SSO requirements for managed team deployments

The important stance is:
- single-user is an early product posture
- multi-user is already a supported architectural shape

That turns the future change into a migration and rollout, not a rewrite of queries, event payloads, service signatures, and UI assumptions.

## 21.5 Authorization model

Keep authorization deliberately simple early, but put the hook points in now.

Start with two roles:
- `owner`
- `operator`

Recommended meaning:
- `owner` manages account settings, secrets, provider configuration, execution targets, token issuance and revocation, and later membership management
- `operator` can create sessions, run tasks, use approved tools, inspect artifacts, and resolve ordinary workflow actions within policy

Rules:
- authorization belongs in application services and policy modules, not scattered across CLI flags or LiveView conditionals
- every service action should receive actor and account context
- approvals, secret management, execution target administration, and destructive admin actions are the first places that should check role
- future RBAC can extend the same policy layer with resource-scoped permissions without changing the outer product shape

Defer the full RBAC matrix to a later phase, but do not defer the existence of:
- explicit roles on identity records
- actor/account context in service calls
- policy checks at the service boundary
- audit metadata on important changes


## 22. Secrets management
Treat secrets as runtime inputs, not ordinary settings.
Prefer ephemeral resolution over persistence.
Persist only what is operationally necessary and only behind one explicit boundary.

## 22.1 Sources and precedence

Resolve secrets in this order:
1. environment variables
2. config file values
3. database-backed secrets

Rules:
- env vars win because they are easiest to inject per deploy, per target, and per process
- config files are for local operator convenience, not shared secret truth
- DB-backed secrets are the last resort for the small set of credentials the product itself must reuse
- missing required secrets should fail fast at startup or before the relevant operation begins
- request payloads, prompts, session transcripts, artifacts, and event history must never become fallback secret stores

Never persist:
- raw user-entered API keys unless the feature explicitly requires server-side reuse
- one-time bootstrap tokens
- OAuth device codes, browser auth codes, or short-lived exchange material
- decrypted secret values outside process memory

## 22.2 Storage at rest

Most secrets should stay out of the database.
For the one table that legitimately needs stored provider credentials or similar shared integration material, use an optional `Cloak`-encrypted Ecto field.

Storage rules:
- keep encrypted values in a dedicated schema field, not mixed into generic JSON blobs
- store metadata separately: provider, label, scopes, last_rotated_at, revoked_at, created_by_user_id
- make the encryption boundary narrow and obvious
- decrypt only in the service path that needs the value
- design the schema so a second active credential can coexist during rotation

`Cloak` is optional because some deployments will stay env- or keychain-only.
The plan still needs one sanctioned encrypted-column path so shared credentials do not drift into ad-hoc storage hacks.

## 22.3 Rotation and revocation

Rotation must not require downtime.
Support overlap where the new credential is written, validated, promoted, and then the old credential is revoked.

Operational flow:
1. add the replacement secret
2. run a reachability check against the target provider
3. switch new work to the replacement
4. drain or finish in-flight work using the old credential if needed
5. revoke the old credential
6. stamp audit metadata

CLI requirement:
- `<APP> secrets rotate <name>` for managed rotation
- `<APP> secrets revoke <name> --version <id>` for explicit cut-off
- rotation commands must never print raw secret material back to stdout

## 22.4 CLI prompts and local keychain

The CLI should make local secret setup boring and safe.

Default flow:
- `<APP> secrets set <name>` prompts without echo
- the CLI stores the value in the OS keychain or credential manager by default
- `<APP> secrets rm <name>` removes the local value and any cached metadata
- `--stdin` or `--from-file` may exist for automation, but the interactive prompt is the default human path
- config-file fallback is allowed only when keychain support is unavailable, with strict file permissions and a loud warning

Do not teach operators to paste secrets into shell history, plain-text notes, or long-lived dotenv files unless they explicitly choose that trade-off.

## 22.5 Doctor checks

`<APP> doctor` should verify secret health without revealing the secret.

Checks:
- presence: is the required secret resolvable from the expected source
- format: does it match the expected prefix, length, or parse shape
- reachability: can the dependent provider or service authenticate with it
- provenance: is the active source env, keychain/config, or DB-backed secret

Output rules:
- print pass/fail and remediation hints
- print source type, not secret value
- never print full tokens, decrypted values, or headers
- if a provider returns an auth error, surface the error class and request id when available, not the submitted credential

## 22.6 Redaction discipline

Redaction is mandatory, centralized, and non-optional.

Apply the same redactor to:
- telemetry fields
- structured logs
- event envelopes
- artifacts
- approval payloads
- LiveView assigns and pushed payloads
- crash reports and diagnostic bundles

Rules:
- keep the redaction list in code, reviewed like any other security-sensitive logic
- do not make secret-detection patterns runtime-configurable by end users
- redact by key name, schema field, and known token formats
- prefer replacing with stable markers such as `[REDACTED]` rather than partial echoes
- any subsystem that emits externalized data should call the central redactor before serialization or persistence


## 23. Dependency choices / frozen stack
## 23.1 Principles

Freeze a small, boring, well-maintained stack.
Prefer Elixir stdlib, OTP, Phoenix, and LiveView built-ins before adding libraries.
Do not add meta-frameworks that hide control flow, error handling, or transport details.
Every dependency must justify its maintenance cost, upgrade churn, operational risk, and debugging footprint.
If a problem can be solved cleanly with a small local module, prefer the local module.

## 23.2 HTTP client

Use `Req` for outbound HTTP.
Do not standardize on `HTTPoison`, and do not expose `Finch` directly as the application-facing client.

Why:
- ergonomic and modern API
- composable request pipeline
- good fit for retries, headers, auth, and response shaping
- one client surface for providers, remote execution targets, and webhooks

Rule:
- all outbound HTTP goes through one internal boundary built on `Req`

## 23.3 LLM provider adapters

Use custom thin adapters.
Do not adopt `instructor`, `langchain`, or provider SDK wrapper stacks as the core abstraction.

Shape:
- one module per provider family
- OpenAI-compatible adapter
- Anthropic-compatible adapter
- local-model adapter
- each adapter implements one small shared behaviour

Rationale:
- keep the contract explicit
- keep payload mapping inspectable
- keep provider quirks isolated
- avoid framework lock-in and leaky abstractions

## 23.4 Testing

Use `Mox` for behaviour-based mocks.
Use fixture files for provider responses, based on captured real payloads and checked into version control.
Use `StreamData` for property tests where invariants matter.
Use `ExUnit` built-ins for everything else.

Rules:
- mock behaviours, not concrete modules
- prefer real payload fixtures over hand-written fantasy JSON
- version fixtures when provider payloads change
- keep test helpers thin and obvious

## 23.5 JSON

Use `Jason`.
It is the de facto Elixir standard, well understood, and good enough.
Do not chase novelty here.

## 23.6 Graph visualization

Use `Cytoscape.js`, wired into Phoenix LiveView through a JS hook.

Why:
- mature library
- handles large graphs well
- strong layout algorithms
- good interactive graph ergonomics

Rule:
- do not reinvent graph rendering with ad-hoc SVG code unless the product proves Cytoscape is insufficient

## 23.7 CLI distribution

Use `Burrito` to ship the CLI as a single cross-platform binary with the BEAM bundled.
The host should not need a preinstalled Elixir or Erlang runtime.
That keeps operator setup simple and reduces environmental drift.

## 23.8 Frozen list + upgrade discipline

Maintain one canonical `mix.exs` dependency list and lock it.
Upgrades go through a review checklist with rationale, release-note scan, compatibility check, and rollback plan.
Do not casually run `mix deps.update` and commit whatever falls out.
Dependency churn is a product decision, not background noise.


## 24. Testing and quality strategy
Testing is a product boundary, not cleanup work.
Use the cheapest test that can prove the behavior, keep the fast layers dominant, and keep full-stack tests rare and valuable.
Mock only at true external boundaries.

## 24.1 Test pyramid

Use a deliberate pyramid:
- unit -> integration -> property -> contract -> LiveViewTest -> CLI smoke -> E2E
- roughly 55-65% unit tests
- roughly 20-25% integration tests
- roughly 5-10% property tests
- roughly 5-10% contract tests
- roughly 5-10% LiveView tests
- a handful of CLI smoke tests per command family
- a tiny E2E suite with only the highest-value journeys

Rules:
- prefer the lowest layer that can prove the rule
- do not push domain behavior upward into brittle UI or E2E coverage
- every escaped bug should earn the cheapest permanent test that would have caught it

## 24.2 Unit tests

Use `ExUnit` for:
- pure functions
- context modules
- application services with boundaries stubbed by behaviour
- validation, policy, and state-transition logic
- serializers, normalizers, and projector helpers

Rules:
- default to `async: true`
- keep fixtures small and intention-revealing
- do not touch the database unless the behavior is actually about persistence
- keep assertions focused on one behavior at a time

## 24.3 Integration tests

Integration tests should hit the real local runtime shape:
- real Ecto repo with SQL sandbox
- real schema constraints and transactions
- real OTP supervision trees where lifecycle matters
- real PubSub, registries, and background-process wiring where applicable

Mock only at external boundaries:
- LLM providers
- outbound HTTP APIs
- remote execution targets not available in CI

Rules:
- do not mock contexts, repos, or runtime internals in this layer
- test restart, subscription, and concurrency behavior where the product depends on it
- do not hide race conditions behind arbitrary sleeps

## 24.4 Property tests

Use `StreamData` where invariants matter more than examples.
Priority targets:
- event envelope invariants
- session/task/turn state machines
- redaction never leaking secrets or partial tokens
- command/event encode-decode round-trips
- projection idempotence where replay matters

Rules:
- generators should reflect real domain shapes, not random nonsense
- any found counterexample should become a named regression test
- use properties for invariants and state spaces, not as ceremony

## 24.5 Contract tests

The runtime behaviour and application service contract are public boundaries.
Every implementation must pass the same shared contract suite.

Verify at least:
- the native runtime backend
- any future Jido runtime backend
- any alternative application service implementation introduced for isolation, testing, or packaging

Rules:
- keep one shared contract test module per contract
- run the same examples against every implementation
- update the contract tests before expanding the interface
- do not let one implementation grow secret behavior outside the contract

## 24.6 LiveViewTest

Every LiveView route must have at least:
- one smoke test that mounts and renders
- one interaction test that exercises the primary operator action

The approvals flow must be fully covered:
- queue rendering
- detail and diff display
- approve path
- reject path
- cancellation or expiry handling
- follow-up state updates in the session/timeline UI

Rules:
- test rendered behavior, events, and navigation
- do not overfit tests to private assigns
- keep JS-hook logic thin so `LiveViewTest` remains the main coverage tool

## 24.7 CLI smoke tests

Spawn the shipped `Burrito` binary in release-like CI jobs.
In development, the same smoke scenarios may run through `mix escript` or the equivalent dev packaging path for speed.

Cover a golden path for each command family:
- help/version/bootstrap
- `doctor`
- config management
- session lifecycle
- task or approval flows once those commands exist

Rules:
- assert exit status, key output markers, and durable side effects
- keep output assertions resilient to harmless formatting drift
- do not depend on an already-running developer shell session or hand-prepared machine state

## 24.8 End-to-end tests

Keep E2E tests intentionally small in number.
Drive them via the CLI against a local Phoenix instance with:
- real Postgres
- real application supervision
- faked LLM/provider fixtures
- deterministic test data setup

Use E2E for only the flows that justify the cost:
- bootstrap or login
- session creation and attach
- one prompt -> tool -> approval -> completion path
- one failure -> diagnosis -> recovery path when repair lands

Rules:
- run serially
- forbid real outbound provider calls
- assert durable records, events, and artifacts, not just terminal text
- if a behavior is already proven below, do not duplicate it here without a reason

## 24.9 Fixture strategy

Use captured real provider responses as fixtures.
Do not rely on hand-written fantasy payloads unless the provider truly cannot supply a stable capture.

Fixture rules:
- store fixtures per provider, model family, API shape, and streaming/non-streaming variant
- keep them versioned in git
- record capture metadata such as provider, model, API date/version, and redactions applied
- preserve streaming chunk order when streaming behavior matters
- replay them through `Mox` at the provider boundary
- review fixture diffs like code

Update rules:
- fixture refresh is an explicit command, not a test side-effect
- tests must never silently rewrite fixtures
- when provider payloads change, add or version fixtures intentionally and update contract expectations with them

## 24.10 Coverage and quality gates

Set explicit quality floors.
Recommended targets:
- 85%+ coverage on contexts and application services
- 70%+ overall project coverage
- higher practical coverage on security-, policy-, and redaction-critical modules

CI must fail if any of these fail:
- `mix format --check-formatted`
- `mix compile --warnings-as-errors`
- test suite with coverage enabled
- `mix credo --strict`
- `mix dialyzer`

Rules:
- all warnings are errors in CI
- flaky tests are defects, not background noise
- coverage is a floor, not permission to skip meaningful assertions


## 25. Observability and telemetry baseline
Observability is part of the runtime contract, not garnish.
Stage 1 ships with logs, metrics, telemetry, and redaction already wired in.
If the system is not measurable, it does not ship.

## 25.1 Stage 1 non-negotiable

Do not promise to “add observability later.”
Later is how teams end up debugging folklore instead of systems.

Minimum bar for Stage 1:
- every important state transition emits telemetry
- every external call is timed and attributed
- every failure has structured context
- every operator-visible action can be traced by request, session, and turn

## 25.2 `:telemetry` taxonomy

Use one canonical event shape:
`[:<app>, <context>, <action>, :start|:stop|:exception]`

Rules:
- keep `<context>` stable and boring
- keep `<action>` verb-oriented and specific
- do not invent near-duplicate names in random modules
- use `:start`, `:stop`, and `:exception` consistently so timings and failures aggregate cleanly

Core contexts and expected actions:
- `session`: create, resume, archive
- `turn`: start, stream, complete, cancel
- `task`: enqueue, start, retry, complete
- `tool_run`: dispatch, stream, complete, fail
- `artifact`: write, promote, reject
- `approval`: request, grant, deny, expire
- `provider_call`: request, stream, complete, fail
- `budget`: snapshot, threshold_crossed, exceeded

## 25.3 Structured JSON logs

Emit structured JSON logs in every environment.
One event per line.
No prose soup.

Required fields:
- `ts`
- `level`
- `event`
- `request_id`
- `session_id`
- `turn_id`
- `actor`
- `msg`

Implementation rules:
- use a JSON logger formatter/backend, not ad-hoc string interpolation
- keep event names aligned with the telemetry taxonomy
- attach IDs from the current request/runtime context, not whatever a caller remembered to pass
- apply redaction before emit, not in some downstream log sink fantasy

## 25.4 Metrics (PromEx)

Use PromEx as the metrics spine.
Enable built-in plugins for Phoenix, Ecto, Oban, and the BEAM VM.
Add one custom plugin for `<app>` domain events so sessions, turns, tasks, tool runs, approvals, provider calls, and budget thresholds all show up as first-class metrics.

Dashboards are code:
- check Grafana dashboards into the repo
- version them with the app
- review dashboard changes like any other operational contract

## 25.5 Tracing hook points

Be OpenTelemetry-ready from the start.
Every runtime kernel command and every emitted runtime event carries `trace_id` and `span_id`.
Provider calls, tool runs, approvals, remote target operations, and repair flows should open spans at the service boundary and propagate them through downstream work.

Tracing backend stance:
- optional in local dev
- required in staging
- required in production

## 25.6 Redaction rules

Keep one redactor module as the single source of truth.
It owns the scrub list for field names and known sensitive shapes.
At minimum, scrub API keys, bearer tokens, cookies, prompts containing user data, raw tool input, raw tool output, and provider-specific credential fields.

Apply the same redactor to:
- logs
- telemetry payloads
- event envelopes
- LiveView streams
- diagnostic bundles

Rules:
- do not duplicate redaction logic across subsystems
- do not let individual callers opt out
- prefer stable replacements such as `[REDACTED]`
- redact before serialization, persistence, broadcast, or export

## 25.7 Cost and budget visibility

Every provider call emits cost telemetry.
That means tokens in, tokens out, model, provider, latency, and USD estimate.
The budget context listens to those events, maintains running totals, and emits threshold-crossed and budget-exceeded events.

Cost visibility is not a later analytics project.
It is how the runtime avoids acting surprised by its own bill.

## 25.8 Health and readiness

Expose `/healthz` for liveness and `/readyz` for readiness.
`/readyz` should fail when the system cannot safely take work, not only when the VM is technically still breathing.

Check at least:
- database connectivity
- PubSub availability
- Oban readiness when enabled
- provider configuration and reachability
- critical execution target reachability

`<APP> doctor` must exercise the same checks through the same modules.
One truth. Two surfaces.


## 26. CI/CD and release engineering
Releases must be boring, repeatable, and auditable.
No heroics, no laptop-only release rituals, no mystery artifacts.

## 26.1 CI pipeline (GitHub Actions)

On every pull request, GitHub Actions must run:
- checkout
- `erlef/setup-beam`
- cache `deps` and `_build`
- `mix deps.get`
- `mix compile --warnings-as-errors`
- `mix format --check-formatted`
- `mix credo --strict`
- `mix dialyzer`
- `mix test --cover`
- `mix coveralls.json`
- upload the generated coverage JSON artifact

Rules:
- fail on any warning
- do not keep advisory lint jobs that can fail quietly
- keep one canonical CI workflow, not near-duplicate YAML drift

## 26.2 Quality gates

A change does not merge unless all of these pass:
1. format check
2. Credo in strict mode
3. Dialyzer clean
4. tests green
5. coverage at or above the stated target
6. no `IO.inspect` or `IO.puts` debug leftovers in non-dev code
7. `mix xref unreachable`

The coverage target must live in repo policy, not team folklore.

## 26.3 Release artifacts

Produce three release outputs:
1. backend tarball from `mix release`
2. CLI single-binary builds via `Burrito` for Linux, macOS, and Windows
3. optional Docker image for the backend release

Rules:
- every artifact is versioned from the release tag
- publish checksums
- sign tags and artifacts where practical
- do not hand-build one-off release artifacts outside the pipeline

## 26.4 Docker

The backend Docker path should be release-based, not Mix-based.

Requirements:
- multi-stage Dockerfile
- build with `mix release`
- final image runs the release directly
- run as a non-root user
- use a minimal base image such as `alpine` or `debian-slim`
- keep runtime layers free of secret-bearing build material
- provide a Compose file for local development with Postgres + backend

## 26.5 Versioning and tagging

Use:
- SemVer for the backend
- CalVer for the user-facing CLI

Rules:
- every release tag is signed
- changelog generation is driven from Conventional Commits messages
- compatibility expectations between backend and CLI versions must be explicit
- never rebuild or mutate a released tag in place

## 26.6 Deployment model

Start with a single-node deployment.
That can live on Fly.io, Render, or a boring bare VM.

Scope:
- orchestration is out of scope initially
- the Oban scheduler lives in the same release for now
- splitting schedulers or workers into separate deployables is a later operational decision, not a default assumption

## 26.7 Secrets in CI/CD

CI/CD secrets live in GitHub Actions secrets or environment-scoped equivalents.

Rules:
- scope secrets as tightly as possible
- rotate them
- never echo them in logs
- never bake them into container layers
- prefer short-lived credentials or OIDC where the platform supports it
- build containers must not ship secret-bearing layers forward into release images

## 26.8 Release checklist

Before tagging a release, verify:
- tests are green
- changelog is updated
- `doctor` passes against staging
- backup/restore dry-run is complete
- observability dashboards and alerts are verified



## 27. Stage-by-stage deferral ledger
This ledger tracks v2 architectural pieces that are documented as reference design in earlier sections but whose full enforcement/implementation is deferred to a specific stage. Use this as the single source of truth for "what ships when" on the Swarm-derived machinery.

|---|---|---|---|
| §12.6.3 | Full priority bands (100–0) | Stage 6 | Single-agent UX is genuinely good (Stage 6 go/no-go gate) |
| §12.6.4 | LLM steering evaluator | Stage 6 | Hard deterministic checks in bands 100–90 and 89–80 proven stable in Stage 4/5 |

### Deferral rules

- Deferred pieces MAY be stubbed earlier (e.g., the event schema for `ToolBlocked` is required from Stage 4) but MUST NOT be in the critical path before their target stage.
- The hook contract (§12.6.1) and execution order (§12.6.2) are NOT deferred — they ship from Stage 2 onward as part of the runtime contract.
- The category-narrow-only invariant (§4.5, §10.3) is NOT deferred — it is enforced from Stage 5 (when execute mode ships).
- Blocked-attempts-as-events (§2.7, §11.5, `tool_runs` schema) is NOT deferred — it is required from Stage 4 (first read-only tools).
- The deterministic invariants in §4.5 are not deferred; they are hard rules from the stage at which their governing capability ships.

### Cross-references

- The priority bands reference table is defined in §12.6.3; enforcement scope by stage is described in §17 (Stages 4, 5, 6).
- The steering evaluator contract is defined in §12.6.4; its enablement gate is covered under §17 Stage 6 "Done means done".

### How to update this ledger

When a deferred piece ships (i.e., its target stage closes), add a row note indicating the stage closed and the piece is now active, rather than removing the row. This preserves the historical record of what was deferred and when it landed. New deferrals (added in future revisions) must have a target stage and a named gate, and must reference a section where the deferred piece is fully specified.
